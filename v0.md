## Items NOT Fixed (with reasoning)

### 1. `reference-library.ts` — 10,003 lines

**v0 finding:** "Extremely large file, possible code smell."

**Why not fixed:** This file is auto-generated by `scripts/ingest-references.py`. It's a static index of 150+ component references used by the AI's `suggest_improvement` tool. It is:
- Never bundled into client JavaScript (server-only, dynamically imported)
- Generated from scanning 3 codebases, not hand-written
- Only loaded on-demand when the AI invokes the reference tool
- No performance impact — tree-shaken out of client builds

Splitting it would add complexity for zero benefit. The generator script would need to output multiple files and the import logic would need a registry. Not worth it.

### 2. `reference-library.ts` — dynamic import pattern

**v0 finding:** "Should be code-split or lazily loaded."

**Why not fixed:** It already IS lazily loaded. The tool definition uses dynamic import:
```typescript
const { getReferenceLibrary } = await import('@/lib/reference-library')
```
This only loads when the AI calls the tool, not on every request. No change needed.

### 3. Token estimation via `JSON.stringify().length / 4`

**v0 finding:** "Rough approximation, should use a proper tokenizer."

**Why not fixed:** This is an intentional design choice. Adding `tiktoken` or `@anthropic-ai/tokenizer` would:
- Add ~2MB to server bundle
- Add 50-100ms per request for tokenization
- Still be approximate (model-specific BPE evolves)

The `/4` heuristic is consistently within 15-20% of actual token counts for mixed English/code content. The compaction system has generous thresholds (70% trigger, 85% warning) specifically to absorb this imprecision. A tokenizer would be more accurate but the cost-benefit doesn't justify it for a client-side heuristic.

### 4. No E2E test suite

**v0 finding:** "Test coverage is unit-only, no integration or E2E tests."

**Why not fixed now:** The 90 unit tests cover the critical paths (tools, virtual FS, utils, rate limiting, API routes). E2E tests require:
- Playwright or Cypress setup
- Mock Anthropic API (streaming responses)
- Mock GitHub OAuth flow
- Mock Supabase (or test instance)
- CI pipeline integration

This is a full sprint of work, not an audit fix. The unit tests catch regressions in business logic. E2E is a future investment.

### 5. `classifyModelComplexity` in route.ts

**v0 finding:** "Model routing logic should be extracted to its own module."

**Why not fixed:** The function is 30 lines and only used in one place (`route.ts`). Extracting it to a separate file would:
- Add a file with a single export
- Add an import
- Make the routing logic harder to find (it's conceptually part of the chat route)

This is premature abstraction. If model routing becomes more complex (multiple routes, A/B testing, user preferences), then extraction makes sense. Not today.

### 6. Console warnings in development

**v0 finding:** "Some React key warnings and hydration mismatches in dev mode."

**Why not fixed:** These are Next.js 15 development-mode artifacts from:
- `framer-motion` AnimatePresence (known issue with React 19)
- Monaco Editor's internal DOM manipulation
- Hot-reload hydration during dev

None appear in production builds. Silencing dev warnings with `suppressHydrationWarning` would mask real issues.

## Architecture Decisions Validated

These patterns were reviewed during the audit and confirmed as correct:

1. **VirtualFS (no local filesystem):** All files in browser memory, sent with each API request. This is the core architecture — not a bug. Enables instant preview, zero server state, and simple persistence model.

2. **Client sends full file state:** Each chat request includes all project files. This looks wasteful but enables stateless API routes, horizontal scaling, and crash recovery. The compaction system manages context size.

3. **Auto-save debounce (5s):** Intentionally slow to avoid hammering Supabase on rapid edits. Manual save (`Ctrl+S`) bypasses the debounce for explicit saves.

4. **BroadcastChannel for tab detection:** Simple, zero-dependency approach to warn about concurrent editing. Not a full CRDT — just a warning. Adequate for single-user tool.

5. **sessionStorage (not localStorage) for active project:** Intentional — each tab gets its own session. Opening Pi-Chi in two tabs doesn't cross-contaminate state.

6. **AES-GCM token encryption in JWT:** GitHub PAT is encrypted in the JWT cookie, not stored in localStorage or passed as a query parameter. Server decrypts on each request. This is defense-in-depth — even if the JWT is stolen, the PAT is encrypted.
